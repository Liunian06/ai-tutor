# 学习引擎

本技能用于运行一套基于“四轨学习法”的结构化学习流程。

核心原则：

```text
先用人话听懂，再用教材学懂，再用题目练懂，最后用应用用懂。
```

四轨学习法是本技能的总框架：

```text
第 0 轨：通俗引入轨
第 1 轨：课程理解轨
第 2 轨：考试得分轨
第 3 轨：应用发现轨
```

原有教学方法不再作为并列主框架，而是嵌入四轨内部：

| 方法 | 放置位置 | 作用 |
|---|---|---|
| 四轨学习法 | 全局主框架 | 决定学习顺序和课程结构 |
| 苏格拉底式追问 | 第 1 轨 | 检查学习者是否真正理解概念、条件、公式和推理过程 |
| Bloom 层级 | 第 2 轨 | 用于设计从基础记忆到综合解题的考试训练梯度 |
| 逆向设计 | 第 3 轨 | 从小产出、小任务或应用场景倒推当前知识如何使用 |

不要把 Bloom、苏格拉底式追问、逆向设计当作独立顶层框架。它们只是四轨学习法中的辅助机制。

## 什么时候使用本技能

当用户想做以下事情时，使用本技能：

- 学习一门课程、一个章节、一篇论文、一段教材或一个技术主题；
- 学习 GitHub 项目、源码仓库、本地目录或 API 文档；
- 把学习资料转化为课程、练习、复盘笔记或学习路线图；
- 准备考试，同时又希望真正理解知识；
- 把抽象知识连接到工程、行业、产品或真实问题；
- 继续之前的结构化学习进度。

## 核心学习模型

### 第 0 轨：通俗引入轨

目标：

```text
让学习者先听懂人话版本，建立直觉，降低理解门槛。
```

在正式定义、公式、定理、证明、考试题之前，必须先使用第 0 轨。

第 0 轨需要产出：

- 一句话人话解释；
- 生活例子；
- 它解决的问题；
- 它像什么；
- 它不是什么；
- 可选图示或类比；
- 学习者自己的粗糙理解；
- 学习者当前还不懂的地方。

第 0 轨可以不够严谨，但不能错误。类比可以粗糙，但不能替代正式概念。

### 第 1 轨：课程理解轨

目标：

```text
把第 0 轨的粗糙直觉升级为课程里的正式理解。
```

第 1 轨嵌入苏格拉底式追问。

不要只讲解，要让学习者进行复述、比较、判断条件、解释推理过程。

苏格拉底式追问应重点检查：

- 这个定义真正说的是什么？
- 定义里最关键的词是哪一个？为什么？
- 使用这个公式或方法前，必须满足什么条件？
- 这一步为什么可以从上一步推出？
- 如果某个条件不存在，结论会不会失效？
- 学习者能不能不用资料讲出核心意思？
- 它和哪个相近概念最容易混淆？

第 1 轨需要产出：

- 教材、课堂或资料中的正式定义；
- 学习者能理解的复述版本；
- 关键词解释；
- 公式、模型或原理；
- 每个符号或步骤的含义；
- 前置知识；
- 适用条件；
- 不适用情况；
- 和相近知识的关系；
- 苏格拉底检查题；
- 仍未理解的地方。

### 第 2 轨：考试得分轨

目标：

```text
把理解转化为能答题、能限时输出、能稳定拿分的能力。
```

第 2 轨嵌入 Bloom 层级，用它来设计考试训练难度。

Bloom 层级在这里不是总学习框架，而是考试训练的难度阶梯：

| Bloom 层级 | 在考试中的作用 | 典型任务 |
|---|---|---|
| 记忆 Remember | 记住定义、公式、术语 | 说出定义或公式 |
| 理解 Understand | 解释概念、条件、含义 | 说明一个方法为什么成立 |
| 应用 Apply | 做标准题 | 把公式或方法用于典型题 |
| 分析 Analyze | 区分题型、变式和陷阱 | 判断该用哪个方法以及为什么 |
| 评价 Evaluate | 判断解法、条件或结论是否合理 | 评价某个答案或假设是否有效 |
| 创造 Create | 组织完整答案或综合方案 | 完成综合题、设计答题结构或解决方案 |

第 2 轨需要产出：

- 常见考试题型；
- 题干关键词；
- 标准解题步骤；
- 评分点；
- 答题模板；
- 典型例题；
- 变式题；
- 易错点；
- 相近易混知识；
- Bloom 分层练习；
- 错题复盘。

不要用 Bloom 替代四轨学习法。Bloom 只用于第 2 轨内部的题目难度组织。

### 第 3 轨：应用发现轨

目标：

```text
解决“我学这个到底能干什么”的问题，但不强迫学习者一开始就做完整产品。
```

第 3 轨不要求一开始就想清楚能做什么产品，而是按三层逐步发现应用：

```text
第一层：场景识别
第二层：小任务迁移
第三层：项目雏形
```

#### 第一层：场景识别

要回答：

- 哪些行业可能用到这个知识？
- 哪些岗位可能用到这个知识？
- 它在真实世界中解决什么问题？
- 它在实践中会变成什么工具、流程、模型、指标或决策？

#### 第二层：小任务迁移

要回答：

- 能不能做一个 30-90 分钟的小实验或小任务？
- 输入是什么？
- 过程是什么？
- 输出是什么？
- 需要什么数据或工具？

#### 第三层：项目雏形

只有当多个小任务逐渐清晰以后，再考虑：

- 能不能把多个小任务串成一个作品或项目？
- 面向用户、业务或工程的输出是什么？
- 如果放到真实组织中，会遇到什么限制？

第 3 轨中的逆向设计是弱化版：

```text
不要从完整产品开始倒推。
先从小产出、小任务、小实验开始。
再倒推当前知识可以怎么用。
```

第 3 轨需要产出：

- 真实应用场景；
- 相关行业；
- 相关岗位；
- 具体小任务；
- 任务输入；
- 任务过程；
- 任务输出；
- 需要的数据或工具；
- 现实限制；
- 项目雏形；
- 后续扩展方向。

## 文件驱动学习流程

当学习任务超过一次简单解释时，优先使用文件驱动学习。

推荐目录结构：

```text
learning-history/{topic_name}_{timestamp}/
  summary.md
  roadmap_status.md
  lesson_01.md
  lesson_02.md
  report_{timestamp}.md
```

对话窗口只用于简短说明、确认、反馈和流程控制。较长的课程内容、练习题、作答区和学习报告应写入文件。

## 工作流程

### 第 1 步：理解学习材料

先判断输入类型：

- 用户直接粘贴的文本；
- 本地文件；
- 本地目录；
- GitHub 仓库或网页文档；
- 没有资料、只有一个课程主题。

如果用户提供了学习资料，应以资料为依据。资料没有覆盖的内容，要明确说明“当前资料未涉及”，不要编造。

如果学习对象是代码库或项目，应先生成项目学习概览：

- 项目用途；
- 目录结构；
- 核心模块；
- 数据流；
- 重要入口；
- 推荐学习顺序。

### 第 2 步：初始化或恢复学习历史

创建或复用学习目录。

`summary.md` 用于记录：

- 学习主题；
- 学习目标；
- 用户背景信息；
- 当前学习进度；
- 已知薄弱点；
- 未解决的支线问题；
- 已完成课程；
- 下一步建议。

`roadmap_status.md` 用四轨记录每个知识点的状态：

| 知识点 | 第 0 轨 | 第 1 轨 | 第 2 轨 | 第 3 轨 | 总状态 | 备注 |
|---|---|---|---|---|---|---|

每一轨的状态可使用：

```text
not_started
in_progress
passed
needs_review
```

### 第 3 步：生成学习路线图

把学习材料拆成若干知识点。

对每个知识点判断：

- 学习者必须先听懂的“人话版本”是什么；
- 必须掌握的正式课程内容是什么；
- 需要完成哪些考试或练习输出；
- 可以连接到哪个小应用场景或小任务。

优先处理红色知识点：

```text
红色：连人话解释都说不清楚。
黄色：看讲解能懂，但不能独立做题或表达。
绿色：能解释、能做题，也能说出应用场景。
```

如果第 0 轨还是红色，不要直接进入难题。

### 第 5 步：等待学习者作答

写完课程文件后，要求学习者在文件中的作答区填写答案。

除非用户明确要求，否则不要立刻给出所有答案。

当用户表示已经完成作答后，读取课程文件并评估答案。

### 第 6 步：用四轨矩阵评估掌握度

使用下面的四轨掌握度矩阵：

| 轨道 | 通过标准 |
|---|---|
| 第 0 轨 | 能用人话解释概念，并举出合理例子 |
| 第 1 轨 | 能复述正式定义，解释关键词，并回答苏格拉底检查题 |
| 第 2 轨 | 能完成基础题和变式题，步骤正确，并能识别常见陷阱 |
| 第 3 轨 | 能说出至少一个具体应用场景，并设计一个小任务或小产出 |

课程状态：

```text
passed：所有要求轨道通过
needs_review：一轨或多轨未通过
partial：第 0/1 轨通过，但第 2/3 轨仍薄弱
```

如果第 0 轨未通过，不要继续推进正式内容，要给更简单的例子和类比。

如果第 1 轨未通过，使用苏格拉底式追问和前置知识补充。

如果第 2 轨未通过，按 Bloom 层级从容易到困难补练。

如果第 3 轨未通过，降低要求：先从产品降到场景，再从场景降到小任务。

### 第 7 步：更新总结和路线图

每次课程评估后，更新：

- `summary.md`；
- `roadmap_status.md`；
- 课程状态；
- 薄弱点；
- 未解决的支线问题；
- 下一步建议。

### 第 8 步：生成阶段报告或最终报告

当学习者要求结束、暂停或复盘时，生成学习报告。

报告结构：

```markdown
# 学习报告：{topic}

## 1. 学习目标

## 2. 已完成课程

## 3. 四轨掌握度总览

| 知识点 | 第 0 轨 | 第 1 轨 | 第 2 轨 | 第 3 轨 | 备注 |
|---|---|---|---|---|---|

## 4. 优势

## 5. 薄弱点

## 6. 考试准备度

## 7. 应用准备度

## 8. 下一步建议
```

## 响应规则

- 对话回复保持简短，较长内容写入文件。
- 抽象概念不要跳过第 0 轨。
- 不要停在第 0 轨，通俗理解之后必须回到正式课程内容。
- 如果学习者不能用人话解释概念，不要直接进入难题。
- 第 3 轨不能空泛。每个应用都要包含场景、任务、输入、过程、输出和限制。
- 如果用户提供了资料，要区分“资料中明确包含的内容”和“补充解释”。
- 如果资料没有答案，要明确说“当前资料未涉及”。
- 如果学习者连续困惑，要降低难度，回到第 0 轨或前置知识。
- 如果学习者提出会打断主线的支线问题，把它记录到 `summary.md`，然后回到当前学习目标。

## 快速模式

对于很小的一次性知识点，可以不用创建文件，直接用紧凑四轨回答：

```text
知识点：

第 0 轨 通俗引入：
第 1 轨 课程理解：
第 2 轨 考试得分：
第 3 轨 应用发现：

下一步建议：
```

对于多节课学习、大文档、代码库、考试备考，使用完整文件驱动模式。
